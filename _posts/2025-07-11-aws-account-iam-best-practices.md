## Introduction
AWS 환경을 효과적으로 운영하려면 계정(Account) 구조와 권한 관리 체계(IAM)에 대한 명확한 이해가 필수적이다.

AWS Account는 리소스 격리, 비용 추적, 보안 정책 적용의 기본 단위이다. IAM(Identity and Access Management)은 사용자, 역할, 정책을 통해 세밀한 접근 제어를 가능하게 한다.

## AWS Account의 기능과 구조적 특징

AWS Account는 AWS와의 **공식적인 비즈니스 관계**를 나타내며, **리소스 생성 및 관리, 액세스 제어, 요금 청구의 기준 단위**이다. 각 계정은 고유한 ID를 가지며, 기본적으로 다른 계정과의 리소스 공유는 허용하지 않는다.

### AWS Account의 주요 기능

AWS 계정은 AWS 서비스 이용에 필수적인 요소이다. 다음 그림과 같이 AWS 계정은 두 가지 주요 기능을 수행한다.

![image.png](/assets/img/posts/aws-account-iam-best-practices/image.png)

#### 1. 리소스 컨테이너 (Resource Container)

AWS 계정은 Amazon EC2, S3, RDS 등 모든 리소스의 **기본 소유 단위**이다.

모든 리소스는 Amazon Resource Name(ARN)에 해당 계정의 ID가 포함되어 고유하게 식별되며, 이는 리소스 추적 및 관리의 기준이 된다.

#### 2. 보안 경계 (Security Boundary)

AWS 계정은 **IAM 사용자 및 역할**, 그리고 권한 정책을 기반으로 인증과 권한 부여를 수행한다.

각 계정은 독립된 보안 도메인으로 작동하며, 계정 간에는 기본적으로 리소스 접근이 불가능하여 보안 사고의 영향을 **계정 단위로 제한**할 수 있다.

### AWS Account의 핵심 기능 및 역할

AWS 환경을 효율적으로 운영하고 확장하기 위해서는 **여러 AWS 계정을 사용하는 다계정 전략(Multi-Account Strategy)**이 모범 사례로 권장된다. AWS 계정은 단순한 인증 수단을 넘어, 다음과 같은 **핵심 기능과 역할**을 수행한다.

#### 1. 비용 모니터링 및 제어

AWS 계정은 **비용이 할당되는 기본 단위**이다.

이를 통해 서로 다른 사업부, 워크로드, 프로젝트 별로 **비용을 명확하게 구분**하고, 클라우드 사용량을 보다 쉽게 **추적, 예산 책정, 예측 및 리포팅**할 수 있다.

- 계정 별 비용 리포팅 외에도, AWS Organizations를 통해 **여러 계정의 비용을 통합 관리**할 수 있는 기능을 제공한다.
- 또한 **서비스 할당량(Service Quotas**) 를 활용하여 과도한 리소스 사용이나 예상치 못한 비용 초과를 방지할 수 있다.

#### 2. 격리 단위로서의 계정

AWS 계정은 **보안, 액세스, 청구의 경계(boundary)** 역할을 하며, 리소스 간의 논리적 격리를 제공한다.

- 각 계정은 자체적으로 리소스를 운영하며, **다른 계정의 리소스와 기본적으로 격리**되어 있어 보안상 유리하다.
- 이를 통해 하나의 계정에서 발생한 **구성 오류, 보안 사고, 악성 작업** 등이 다른 계정의 워크로드에 영향을 미치는 것을 방지할 수 있다.

#### 2. 비즈니스 워크로드 분리

AWS 계정을 사용하여 **공통의 비즈니스 목적을 가진 워크로드를 고유한 계정에 배치**하면, 다음과 같은 이점이 있다.

- 워크로드 별로 **소유권과 의사 결정권을 명확히 분리**할 수 있어 책임 소재가 분명해진다.
- 계정 간 종속성과 충돌을 줄여 **보안성과 운영 안정성**을 높일 수 있다.
- 이러한 구조는 추후에 조직의 사업부 분리, 자회사 매각 등에도 유연하게 대응할 수 있다.

## IAM 사용자, 그룹, 역할, 정책의 구성 방식

IAM은 사용자(User), 그룹(Group), 역할(Role), 정책(Policy)으로 구성된다. 본문에서는 각 요소의 개념과 이들 간의 구성 방식에 대해 살펴보겠다.

### 🆔 IAM 사용자(IAM User)

#### 1. IAM 사용자란?

- IAM 사용자는 AWS 리소스에 접근하는 **사람 또는 워크로드**를 대표하는 주체이다.
- 이름과 자격 증명으로 구성되며, 각 사용자에게 개별 권한을 부여할 수 있다.

#### 2. 사용자 식별 방식

AWS는 IAM 사용자를 다음 세 가지 방식으로 식별한다.

- **사용자 이름**: 콘솔에서 식별 가능한 이름 (예: `John_Doe`)
- **ARN (Amazon Resource Name)**: AWS 전역에서 고유하게 식별

*예시) arn:aws:iam::123456789012:user/John_Doe*

- **고유 ID**: API/CLI 사용 시 반환 되는 내부 식별자

#### 3. 사용자 인증 방식

IAM 사용자는 다음과 같은 방식으로 AWS에 접근한다.

- **콘솔 암호**: AWS Management Console 로그인 용도
- **액세스 키(Access Key)**: CLI, SDK 등 프로그래밍 방식 호출 용도
- **SSH 키**: CodeCommit 접속 용도
- **서버 인증서**: AWS Certificate Manager(ACM)가 지원되지 않는 리전에서 HTTPS 인증 용도

보안 강화를 위해 **MFA(다중 인증)** 사용이 권장된다.

#### 4. 자격 증명 관리 항목

- 암호/액세스 키 생성 및 변경
- MFA 디바이스 등록
- 사용 기록 확인 (마지막 사용 시점 등)
- 불필요한 자격 증명 제거
- 자격 증명 보고서 다운로드 (CSV)

#### 5. 권한 부여 방식

- IAM 사용자 생성 시에는 기본적으로 아무 권한도 없다.
- 권한은 다음과 같은 방식으로 부여할 수 있다.
    - **직접 정책 연결**
    - **그룹을 통한 간접 권한 부여**
    - **역할 수임(AssumeRole)을 통한 임시 권한 위임**

고급 제어가 필요한 경우 **권한 경계(Boundary Policy)** 설정 가능

#### 6. 과금 및 계정 연계

- 각 IAM 사용자는 **하나의 AWS 계정**에만 속하며, 해당 계정에 대한 모든 작업은 그 계정으로 청구된다.
- IAM 리소스 수에는 할당량 제한이 존재한다.

#### 7. 서비스 계정으로서의 IAM 사용자

- 애플리케이션이 사용하는 IAM 사용자(서비스 계정)의 경우, 액세스 키를 코드에 직접 포함하지 말고 환경 변수나 AWS SDK를 통해 안전하게 처리해야 한다.
- 모범 사례로는 **IAM 역할 + 임시 자격 증명** 사용이 권장된다.

### 🆔 IAM 사용자 그룹(IAM User Group)

IAM 사용자 그룹은 **IAM 사용자들의 집합**으로, 그룹에 정책을 연결함으로써 **다수 사용자에게 공통 권한을 일괄 적용**할 수 있도록 도와준다.

#### 1. 사용자 그룹의 활용

- 사용자 그룹에 권한을 부여하면, **그룹에 속한 모든 사용자가 해당 권한을 자동으로 상속** 받다.

*예시) `Admins`라는 그룹에 관리자 권한을 부여하면, 그룹에 속한 모든 사용자가 관리자 권한을 갖는다.*

- 새로운 사용자가 입사하거나 업무가 변경될 경우, **그룹에 추가하거나 이동 시키는 방식으로 권한 관리가 용이**한다.

#### 2. 정책 연결 방식

- 그룹은 **아이덴티티 기반 정책**(identity-based policy)을 통해 권한을 받다.
- 그룹 자체는 **인증 주체(Principal)가 아니므로**, 정책에서 직접 연결 대상으로 지정할 수는 없다.
- 그룹은 리소스 기반 정책의 대상이 될 수 없다.

#### 3. 사용자 그룹의 특징

- 하나의 그룹에 **여러 사용자**를 포함할 수 있다.
- 하나의 사용자가 **여러 그룹에 동시에 소속**될 수 있다.
- 그룹은 **중첩될 수 없다**. 즉, 그룹 안에 다른 그룹을 포함할 수 없다.
- 기본적으로 모든 사용자가 자동 포함되는 그룹은 없다. **그룹은 수동으로 생성 및 할당**해야 한다.
- IAM 그룹 수, 사용자 그룹 멤버십 수 등에는 AWS 계정 단위의 할당량(Quotas)이 존재한다.

#### 4. 예시 시나리오

다음 다이어그램은 작은 조직에서 `Admins`, `Developers`, `Test`와 같은 IAM 사용자 그룹을 생성해, **사용자들을 역할 별로 분류**한 예시이다.

![image.png](/assets/img/posts/aws-account-iam-best-practices/image 1.png)

- `Admins`: 관리자 권한 부여
- `Developers`: 개발 환경 접근 권한 부여
- `Test`: 테스트 리소스 접근 권한 부여

각 사용자(Jim, Brad 등)는 고유한 자격 증명을 가지며, 필요에 따라 하나 이상의 그룹에 소속될 수 있다.

### 🆔 IAM 역할(IAM Role)

IAM 역할은 **AWS 리소스에 대한 임시 접근 권한을 제공하기 위한 자격 증명 객체**로, 특정 사용자에 종속되지 않고 다양한 주체가 수임(Assume)할 수 있다. 역할은 IAM 사용자와 유사하지만, 암호나 장기 액세스 키 없이 **임시 보안 자격 증명**을 통해 접근 권한을 제공한다.

#### 1. IAM 역할의 특징

| **항목** | **설명** |
| --- | --- |
| 자격 유형 | IAM 사용자와 유사한 권한 정책을 가진 자격 증명 |
| 인증 방식 | STS를 통해 발급된 임시 보안 자격 증명 (Access Key, Secret Key, Session Token) 사용 |
| 연관 사용자 없음 | 특정 사용자에 고정되지 않음 |
| 수임 주체 | IAM 사용자, 역할, AWS 서비스, 외부 ID 제공자 등 |
| 일반 사용자와 구분점 | 장기 자격 증명 없음, 직접 로그인 불가 |

AWS STS(Security Token Service)는 IAM 역할 수임(AssumeRole)과 같은 작업을 통해 일정 시간 동안 유효한 **임시 보안 자격 증명**을 발급하는 AWS 서비스이다. 이를 통해 장기 자격 증명을 노출하지 않고도 AWS 리소스에 안전하게 접근할 수 있다.

#### 2. 주요 용도

- AWS 서비스 간 권한 위임 (예: Lambda → S3)
- 크로스 계정 접근
- 모바일/웹 앱 인증
- ID Federation (SAML, OIDC)
- 외부 파트너 감사를 위한 위임

#### 3. IAM 역할 구성 요소

| **구성 요소** | **설명** |
| --- | --- |
| 역할(Role) | 권한 정책이 부여된 IAM 엔터티로, 직접 로그인 없이 수임을 통해 사용됨 |
| 권한 정책(Policy) | 역할이 어떤 AWS 리소스에 대해 어떤 작업을 수행할 수 있는지를 정의 |
| 신뢰 정책(Trust Policy) | 어떤 주체가 이 역할을 수임할 수 있는지를 명시 |
| 보안 주체(Principal) | 역할을 수임하는 대상. IAM 사용자, 역할, AWS 서비스, ID 제공자 등이 포함됨 |

#### 4. 역할의 종류

| 종류 | **설명** | 예시 |
| --- | --- | --- |
| 서비스 역할 | 특정 AWS 서비스가 사용자를 대신해 리소스에 접근하도록 허용하는 역할. 사용자가 직접 생성 및 관리.  | Lambda 함수가 S3 객체를 읽기 위한 역할 |
| 서비스 연결 역할 | AWS 서비스가 자동으로 생성하며 해당 서비스가 소유. 사용자는 삭제할 수 없고 권한만 열람 가능. | Amazon ECS, CloudWatch, Organizations 등 |
| 크로스 계정 역할 | 다른 AWS 계정의 사용자 또는 서비스가 수임하여 리소스에 접근할 수 있도록 허용하는 역할.  | 계정 A의 사용자가 계정 B의 S3 버킷에 접근할 때 |

#### 5. 보안 고려 사항

IAM 역할 보안 설계 시 다음 요소를 고려해야 한다.

- 역할 사용 대상 제한: 신뢰 정책에 과도한 와일드카드(*) 사용 금지
- 권한 경계(Permissions Boundary): 역할이 가질 수 있는 권한의 최대 값 제한
- 세션 태그: 태그 기반 권한 제어 및 감사 지원
- 외부 ID 사용: 제 3자에게 안전하게 역할을 위임할 때 사용
- 최소 권한 부여(Least Privilege): 역할에는 필요한 최소 권한만 부여해야 함

#### 6. 리소스 기반 정책과의 차이

| **구분** | **IAM 역할** | **리소스 기반 정책** |
| --- | --- | --- |
| 수임 방식 | 자격 증명을 수임하여 임시 권한을 얻음 | 리소스 자체에 정책을 부여함 |
| 사용 목적 | 다양한 주체에게 제한된 시간 동안 권한 위임 | 특정 리소스에 대한 접근 권한 직접 제어 |
| 예시 | 외부 사용자에게 S3 접근 권한을 위임 | S3 버킷에 정책으로 다른 계정 사용자 허용 |

### 🆔 IAM 정책(IAM Policy)

IAM 정책은 AWS 리소스에 대한 액세스를 제어하기 위한 JSON 형식의 문서이다. 정책은 IAM 자격 증명(사용자, 그룹, 역할) 또는 리소스에 연결되며, 특정 작업을 허용하거나 거부하는 권한을 정의한다.

- 대부분의 정책은 JSON 형식으로 작성된다.
- 정책은 권한을 **직접 부여**하거나, **제한 조건**으로 사용된다.
- AWS는 정책을 기반으로 요청 시 권한을 평가한다.

#### 1. 정책 유형

AWS에서는 총 7가지 유형의 정책을 지원하며, 각 정책은 권한 부여 방식, 연결 대상, 사용 목적에 따라 구분된다.

| **정책 유형** | **연결 대상** | **설명** |
| --- | --- | --- |
| ID 기반 정책 | 사용자, 그룹, 역할 | 자격 증명에 직접 권한을 부여하는 정책. 가장 일반적으로 사용됨 |
| 리소스 기반 정책 | S3, SNS, Lambda 등 리소스 | 리소스 자체에 연결되며, 크로스 계정 액세스를 지원 |
| 권한 경계 | 사용자, 역할 | 자격 증명에 부여할 수 있는 권한의 최대 범위를 제한 |
| SCP(서비스 제어 정책) | AWS Organizations의 계정 또는 OU | 계정 내 IAM 사용자 및 역할의 최대 권한을 제한 |
| RCP(리소스 제어 정책) | AWS Organizations 내 리소스 | IAM 정책을 변경하지 않고 리소스 수준에서 중앙 제어를 가능하게 함 |
| ACL(Access Control List) | S3, VPC 등 일부 리소스 | JSON이 아닌 형식으로 계정 간 리소스 접근을 제어 |
| 세션 정책 | 임시 세션(AssumeRole, STS) | STS 세션 생성 시 전달되어 임시 권한 범위를 제한함 |

#### 2. IAM 정책 평가 방식

AWS는 요청을 평가할 때 다음 순서로 정책을 적용한다.

1. **명시적 거부(Deny)**
어떤 정책이 "무조건 안 된다(Deny)"고 되어 있으면, 다른 정책이 허용하더라도 **무조건 거부된다**.
2. **명시적 허용(Allow)**
정책에 "허용한다(Allow)"고 명시되어 있고, 조건도 만족하면 **요청이 허용된다**.
3. **기본 거부(Default Deny)**: 
정책이 아무것도 없거나, 해당 작업에 대해 허용이 명시되지 않은 경우는 **기본적으로 거부된다**.

또한, 하나의 사용자나 역할에 여러 개의 정책이 연결되어 있더라도,
각 정책의 설명문(Statement)은 서로 ‘또는(OR)’처럼 적용되며, 그중 하나라도 허용이면 허용된다.
하지만 어디든 '명시적 거부'가 있으면 무조건 거부가 우선 된다.

#### 3. 최소 권한 부여 원칙

IAM 정책 작성 시 보안 모범 사례로 최소 권한(Least Privilege)을 적용해야 한다.

- 처음에는 제한된 권한만 부여하고, 필요한 범위로 점진적으로 확장
- 광범위한 와일드카드(*) 사용은 피하고, 가능한 구체적으로 명시
- IAM Access Analyzer, Access Advisor, CloudTrail 등을 활용해 미사용 권한을 탐지하고 제거

#### 4. 정책 문서(JSON) 구조

IAM 정책은 다음과 같은 요소로 구성된다.

| **요소** | **설명** |
| --- | --- |
| `Version` | 정책 언어 버전 (`2012-10-17` 권장) |
| `Statement` | 하나 이상의 설명문(Statement) 배열 |
| `Sid` | 설명문의 식별자 (선택 사항) |
| `Effect` | `Allow` 또는 `Deny` |
| `Principal` | 리소스 기반 정책에서 사용. 접근 주체를 명시 |
| `Action` | 허용하거나 거부할 작업 목록 |
| `Resource` | 정책이 적용될 리소스의 ARN |
| `Condition` | 정책이 적용되는 조건 (예: MFA 필요 등) |

이 정책은 특정 AWS 계정의 루트 사용자에게 S3 버킷 전체 작업을 허용하지만, MFA가 활성화된 세션에서만 유효하다. 모든 주요 요소(`Sid`, `Effect`, `Principal`, `Action`, `Resource`, `Condition`)가 포함된 완전한 예시이다.

```jsx
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowFullAccessWithMFA",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

## 계정 및 권한 관리를 위한 IAM 보안 모범 사례

AWS 환경에서 **신뢰할 수 있는 보안 체계**를 구축하려면, **IAM(Identity and Access Management)** 설정이 핵심이다. IAM은 사용자의 접근을 통제하고, 각종 리소스에 대한 권한을 정밀하게 제어하는 **클라우드 보안의 출발점**이다.

아래는 AWS에서 공식적으로 권장하는 **IAM 보안 모범 사례**와 함께, 다양한 **기업 환경에서의 실제 IAM 활용 사례**를 예시로 정리한 내용이다. 실무 환경에서 자주 발생하는 보안 리스크를 예방하고, 보다 안전하고 유연한 클라우드 운영을 위한 전략 수립에 참고가 될 수 있다.

### IAM 보안 모범 사례

#### 1. 루트 사용자 최소 사용 및 MFA 필수화

- **모범 사례**:
    - 루트 계정은 계정 생성 이후 MFA를 활성화하고, 로그인 외의 모든 작업은 별도 IAM 계정으로 수행한다.
    - 루트 액세스 키는 절대 생성하지 않으며, 존재할 경우 즉시 삭제한다.
- **기업 사례**:
    
    한 스타트업이 루트 사용자로 EC2 인스턴스를 직접 운영하다가, 루트 계정 비밀번호가 유출되어 전체 리소스가 삭제되는 사고가 발생.
    
    이후 루트 계정은 오직 보안 담당 부서에서만 MFA를 통해 접근하도록 제한하고, 운영은 IAM 관리자 계정으로 이관.
    

#### 2. IAM 역할(Role)을 통한 임시 자격 증명 사용

- **모범 사례**: EC2, Lambda, ECS 등 서비스에는 Access Key 대신 IAM 역할을 부여하여 임시 자격 증명을 활용한다.
- **기업 사례**:
    
    보안이 중요한 금융 기관은 백엔드 애플리케이션에 IAM 역할을 부여하고, 개발자 로컬에는 Access Key를 사용하지 않도록 정책화.
    
    이로 인해 자격 증명 유출 사고를 사전에 차단.
    

#### 3. 최소 권한 원칙 적용

- **모범 사례**: 사용자와 애플리케이션에 꼭 필요한 서비스와 작업만 권한으로 부여하며, 점진적으로 커스텀 정책으로 전환한다.
- **기업 사례**:
    
    SaaS 기업이 QA 엔지니어에게 `s3:GetObject` 권한만 부여해 테스트 리소스만 조회 가능하도록 제한함으로써, 운영 리소스 삭제 사고를 예방.
    

#### 4. 정책에 조건(Condition) 활용

- **모범 사례**: 정책에 MFA 사용 여부, 접근 IP, 태그 등 조건(`Condition`)을 추가해 정밀한 제어를 구현한다.
- **기업 사례**:
    
    글로벌 제조사는 IAM 정책에 `aws:SourceIp` 조건을 설정해, 본사 VPN을 통한 접속만 허용.
    
    이를 통해 외부 지역의 무단 접근을 차단하고 지역별 접근 정책을 강화.
    

#### 5. IAM Access Analyzer로 외부 접근 식별

- **모범 사례**: IAM Access Analyzer를 활용해 외부 계정에 노출된 리소스를 탐지하고, 퍼블릭 설정을 방지한다.
- **기업 사례**:
    
    대형 유통사는 Access Analyzer를 통해 수십 개의 S3 버킷 중 외부 공유 설정이 활성화된 항목을 탐지하고, 보안 설정을 일괄 수정하여 정보 유출 리스크를 제거.
    

#### 6. 장기 액세스 키 비활성화 및 주기적 교체

- **모범 사례**: IAM 사용자의 Access Key는 주기적으로 회전하며, 미사용 키는 삭제한다. 장기적으로는 키 없는 인증 방식으로 전환하는 것이 이상적이다.
- **기업 사례**:
    
    한 MSP 기업은 모든 운영 자동화 계정에 대해 키를 사용하지 않고, EC2 Instance Profile 기반 역할 인증만 사용함으로써 자격 증명 유출 가능성을 구조적으로 차단.
    

#### 7. IAM 리소스 정리 및 감사

- **모범 사례**: 미사용 사용자, 역할, 정책 등 IAM 리소스를 정기적으로 정리하고, 이상 징후를 CloudTrail이나 Access Advisor를 통해 감시한다.
- **기업 사례**:
    
    IPO를 준비 중인 기업이 보안 감사를 대비해 IAM 사용자 중 90일 이상 로그인 이력이 없는 계정 및 키를 식별해 정리.
    
    그 결과, 외부 보안 점검에서도 우수 평가를 받음.
    

#### 8. AWS Organizations 및 SCP 기반 거버넌스 강화

- **모범 사례**: AWS Organizations를 활용해 여러 계정을 통합 관리하고, Service Control Policy(SCP)를 통해 권한을 제어한다.
- **기업 사례**:
    
    다국적 제조업체가 개발·테스트·운영 계정을 분리하고, 운영 계정에는 SCP를 통해 `Delete*` 작업을 전면 차단.
    
    실수나 내부 위협으로부터 운영 리소스를 보호함.

## 다중 계정 환경을 위한 AWS 거버넌스 설계 모범 사례

AWS에서 다계정 환경을 안정적으로 운영하기 위해서는, **AWS Organizations를 활용**하여 계정을 **목적과 역할에 따라 논리적으로 분리**하고 이를 **조직 단위(Organizational Units, OU)로 계층화하여 관리**하는 것이 핵심이다.
이러한 구조는 보안, 감사, 운영, 비용 관리, 정책 적용 측면에서 **높은 유연성과 통제력**을 제공한다.

### AWS Organizations란?

**AWS Organizations**는 여러 AWS 계정을 하나의 조직 단위로 통합 관리할 수 있도록 지원하는 서비스이다.

보안, 비용, 거버넌스, 운영 분리를 위해 다중 계정을 사용하는 환경에서 **계정 구조화와 정책 적용의 중심 플랫폼** 역할을 한다.

AWS Organizations의 주요 기능은 다음과 같다.

![image.png](/assets/img/posts/aws-account-iam-best-practices/image 2.png)

| 기능 | 설명 |
| --- | --- |
|  **Add accounts** | 새 AWS 계정을 생성하거나 기존 계정을 초대하여 조직에 통합 |
| **Group accounts** | 목적·기능·환경에 따라 계정을 OU(Organizational Unit)로 분류 |
|  **Apply policies** | SCP(Service Control Policy)를 통해 OU/계정 단위로 권한 상한선 제어 |
|  **Enable AWS services** | AWS Config, CloudTrail, GuardDuty 등과 통합해 중앙에서 보안 및 로깅 설정 |

### OU(Organizational Unit)의 개념 이해

OU는 AWS Organizations에서 **여러 계정을 그룹화하기 위한 논리적 단위**이다.

조직에서 팀, 부서, 환경별로 계정을 나누듯, OU를 사용하면 이러한 구조를 AWS 내에서도 그대로 반영할 수 있다.

OU의 주요 특징은 다음과 같다.

- **계층 구조**: OU는 트리 형태로 구성되어 하위 OU 및 계정을 포함할 수 있다.
- **정책 적용 단위**: OU에 연결된 SCP는 하위 모든 계정에 자동 적용된다.
- **책임 및 접근 제어 단위**: 역할별/부서별로 권한을 분리하고, 일관된 정책 적용 가능

즉, OU는 단순한 분류 단위가 아닌, **멀티 계정 거버넌스를 실현하는 정책과 구조의 경계선**이다.

### 조직 구조 설계를 위한 OU 구성 전략

AWS에서는 다음과 같은 기준 중 하나 이상을 바탕으로 OU를 구성할 것을 권장한다.

- **업무 기능(Function)**: 예) 보안, 인프라, 애플리케이션, 감사
- **환경(Environment)**: 개발(dev), 테스트(test), 운영(prod)
- **사업부/조직 단위(Division)**: 마케팅, 재무, R&D, 해외 지사 등

OU 설계는 조직의 책임 체계와 일치해야 하며, 정책 관리와 비용 추적의 기준으로도 활용된다.

### 권장 OU 구조 예시

**출처: [AWS Prescriptive Guidance – Privacy Reference Architecture](https://docs.aws.amazon.com/ko_kr/prescriptive-guidance/latest/privacy-reference-architecture/organization-account-structure.html)**

아래 그림과 표는 AWS가 제시하는 보안(SRA) 및 개인정보 보호(PRA) 중심의 권장 조직 구조이다.

OU는 기능별로 분리되며, 각 계정은 역할에 따라 배치되어 **보안, 운영, 프라이버시**를 명확하게 구분할 수 있다.

![image.png](/assets/img/posts/aws-account-iam-best-practices/image 3.png)

| OU 이름 | 포함 계정 | 주요 역할 |
| --- | --- | --- |
| **Org Management** | Org Management Account | 조직 생성, SCP 관리, 결제 계정 (운영 리소스 없음) |
| **Workloads OU** | Application Account | 실제 서비스 및 애플리케이션 운영 |
| **Infrastructure OU** | Shared Services Account, Network Account | AD, CI/CD, SSM 등 공통 서비스 및 네트워크 관리 |
| **Security OU** | Security Tooling Account, Log Archive Account | 보안 서비스 통합 운영 및 로그 중앙 수집/보관 |
| **Personal Data (PD) OU** | PD Application Account, Data Account | 개인정보 처리 및 민감 데이터 저장/관리 |

이 구조는 AWS Control Tower 또는 Landing Zone 구현 시 기본 템플릿으로도 활용되며, 보안 및 감사 기능의 통합, 데이터 민감도에 따른 격리, 비용 및 정책 통제의 기반이 된다.

### OU 단위 정책 적용 전략 (SCP 예시)

OU는 정책 적용의 기준선이기도 하며, SCP를 통해 다음과 같은 제한을 설정할 수 있다.

| OU | SCP 적용 예 |
| --- | --- |
| **Workloads OU** | IAM 권한 생성 금지, 리전 제한, 서비스 화이트리스트 |
| **Infrastructure OU** | EC2, VPC, SSM 등 특정 인프라 서비스만 허용 |
| **Security OU** | 읽기 전용 외 계정 접근 차단, 리소스 생성 제한 |
| **PD OU** | 암호화(KMS) 필수, 민감 리소스 태깅 필수, 로깅 필수화 |
